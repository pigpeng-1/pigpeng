保存相关的逻辑

对于一些实时保存的问题，哪些场景适合实时保存，哪些适合点击按钮后保存；

实时保存相关内容

在用户操作过程中，数据发生变化时自动触发后端保存，无需用户手动操作。其核心作用是避免用户数据丢失，提升操作便利性，减少用户的手动操作成本。

1. 文档/内容编辑类场景

这类场景中用户会进行长时间的内容编辑，实时保存可以避免因意外（如浏览器崩溃、电脑死机）导致的内容丢失；

在线文档编辑：如腾讯文档、飞书文档、Google Docs 等，用户在编辑文档时，每输入一段内容或间隔一定时间（如 1-2 分钟），自动将内容保存到后端，确保编辑的内容不会丢失。

2. 表单草稿类场景

用户在填写长表单时，实时保存草稿可以让用户随时中断填写，下次继续填写，提升用户体验：

  • 长表单填写：如用户注册的详细信息填写、简历填写、申请表单填写等，用户在填写过程中，实时保存已填写的内容，下次打开页面时可以恢复之前的填写进度。
  • 分步表单填写：如多步骤的订单填写、报名表单填写，每完成一个步骤后自动保存该步骤的内容，避免用户在步骤切换时丢失数据。

3. 配置项修改类场景

对于简单的、独立的配置项修改，实时保存可以让用户的修改即时生效，无需手动点击保存：

  • 开关类配置：如后台系统的功能开关、通知设置开关、隐私设置开关等，用户点击开关后，实时将开关状态保存到后端，修改即时生效。
  • 简单参数配置：如字体大小设置、主题颜色设置、页面布局设置等，用户修改参数后，实时将配置保存到后端，无需用户手动点击保存。
  • 个性化设置：如用户的个人偏好设置（如语言设置、时区设置），用户修改后实时保存，确保用户下次使用时保持设置的偏好。

4. 聊天 / 消息类场景

聊天消息需要实时保存，确保用户发送的消息和接收的消息都能被持久化存储：

  • 即时聊天消息：如微信、QQ、企业微信等聊天工具，用户发送消息后，实时将消息保存到后端，确保消息不会丢失，并且在不同设备上可以同步查看。
  • 客服聊天记录：如在线客服系统，用户和客服的聊天记录实时保存到后端，方便后续查看和追溯。

5. 数据采集类场景

对于需要持续采集数据的场景，实时保存可以确保数据的完整性和及时性：

  • 实时数据监控：如物联网设备的实时数据采集（如温度、湿度、设备状态），设备将数据实时上传到后端保存，确保数据的实时性和完整性。
  • 用户行为数据采集：如用户在页面上的点击行为、浏览行为，实时将行为数据保存到后端，用于后续的数据分析。

点击保存相关内容

点击按钮保存适合用于需要用户确认修改、涉及重要业务操作、或者数据修改需要经过审核的场景：

1. 重要业务操作类场景

涉及重要业务逻辑的操作，需要用户明确确认后再保存，避免误操作导致的业务问题：

  • 订单提交：如电商平台的订单提交、外卖订单提交，用户确认订单信息后，点击 “提交订单” 按钮，才将订单信息保存到后端，触发后续的订单处理流程。
  • 支付操作：如在线支付、充值操作，用户确认支付金额和支付方式后，点击 “支付” 按钮，才将支付请求提交到后端，完成支付流程。
  • 数据删除操作：如删除用户信息、删除订单、删除文章等，用户点击 “删除” 按钮并确认后，才将删除请求提交到后端，执行删除操作。

2. 复杂表单提交类场景

对于复杂的、需要用户确认所有内容的表单，点击按钮保存可以让用户在确认所有内容无误后再提交：

  • 复杂业务表单：如企业的采购表单、项目申报表单、合同填写表单等，用户填写完所有内容后，点击 “提交” 按钮，才将表单内容保存到后端，确保内容无误。
  • 多字段关联表单：如表单中的字段之间有复杂的关联关系，用户需要确认所有字段的内容都正确后，点击保存按钮提交，避免部分字段填写错误导致的数据问题。

3. 权限敏感类操作场景

涉及权限修改、敏感数据修改的操作，需要用户明确确认后再保存，确保操作的安全性：

  • 权限修改：如后台管理系统的用户权限修改、角色权限修改，管理员修改权限后，点击 “保存” 按钮，才将权限修改保存到后端，避免误修改权限导致的安全问题。
  • 敏感数据修改：如用户的密码修改、银行卡信息修改、身份证信息修改等，用户确认修改的内容后，点击 “保存” 按钮，才将修改的敏感数据保存到后端，确保数据的安全性。
  • 系统配置修改：如服务器配置修改、数据库配置修改等重要系统配置，管理员修改后，点击 “保存” 按钮并确认，才将配置修改保存到后端，避免误修改导致系统故障。

4. 需确认的操作类场景

对于需要用户明确确认的操作，点击按钮保存可以让用户有明确的操作感知，避免误操作：

  • 内容发布：如文章发布、产品发布、活动发布等，用户编辑完内容后，点击 “发布” 按钮，才将内容保存到后端并发布，确保用户确认内容无误后再发布。
  • 数据修改确认：如修改重要的业务数据（如订单金额、用户等级），用户修改后需要点击 “确认修改” 按钮，才将修改的数据保存到后端，避免误修改。

5. 批量操作类场景

对于批量操作的数据，点击按钮保存可以让用户确认批量操作的内容后再提交，避免批量操作错误：

  • 批量数据导入：如批量导入用户信息、批量导入商品信息，用户上传导入文件并确认导入内容后，点击 “开始导入” 按钮，才将数据批量保存到后端。
  • 批量数据修改：如批量修改商品价格、批量修改用户状态，用户选择需要修改的数据并设置修改内容后，点击 “批量修改” 按钮，才将修改内容保存到后端。

前后端校验逻辑

对于后端校验，和前端校验方面的思考；如果是纯前端校验，可以请求的比较频繁一点，如果是后端校验，则最好放在点击提交按钮时校验；

前端校验

在用户输入数据时即时反馈错误，减少无效的请求发送；

1. 格式类校验

用于验证用户输入的数据是否符合指定的格式要求，这类校验可以在前端即时完成，避免用户提交错误格式的数据：

  • 手机号格式校验：验证输入的手机号是否符合 11 位数字的格式，以及是否符合手机号的号段规则。
  • 邮箱格式校验：验证输入的邮箱是否包含 @ 符号，以及 @ 前后的内容是否符合邮箱格式规范。
  • 身份证号格式校验：验证身份证号的位数（18 位）、最后一位的校验码是否正确（仅格式层面，合法性需后端校验）。
  • 银行卡号格式校验：通过 Luhn 算法验证银行卡号的格式是否正确。
  • 日期格式校验：验证输入的日期是否符合 YYYY-MM-DD、YYYY/MM/DD 等指定格式。

2. 必填项校验

验证表单中的必填字段是否已填写，在用户提交表单前即时提示未填写的字段：

  • 用户注册表单中的用户名、手机号、密码字段。
  • 订单提交表单中的收货地址、联系方式字段。
  • 反馈表单中的内容、联系方式字段。

3. 即时反馈类校验

在用户输入过程中即时反馈错误，提升用户体验：

  • 密码强度校验：实时检测用户输入的密码是否符合复杂度要求（如包含大小写字母、数字、特殊字符，长度不少于 8 位），并给出强度提示。
  • 输入长度校验：验证输入的内容是否在指定的长度范围内，比如用户名长度限制为 2-20 位，输入时即时提示超出长度的错误。
  • 实时格式提示：在用户输入手机号、邮箱时，即时提示格式错误，比如输入手机号时输入了非数字字符，立即提示 “请输入正确的手机号”。

4. 简单逻辑校验

处理无需依赖服务器数据的简单逻辑验证：

  • 密码一致性校验：验证 “密码” 和 “确认密码” 字段的内容是否一致，在用户输入时即时提示是否匹配。
  • 数值范围校验：验证输入的数值是否在指定范围内，比如年龄输入框限制为 1-120 岁，输入超出范围的数值时即时提示。
  • 单选 / 复选框校验：验证是否选择了指定的选项，比如注册时是否同意用户协议。

5. 防重复提交的前端处理

在用户提交表单时，临时防止重复请求：

  • 按钮禁用：点击提交按钮后，禁用按钮，防止用户重复点击提交，适用于简单的表单提交场景。
  • 请求锁定：在请求发送过程中，锁定提交操作，直到请求完成（成功或失败）后解锁。

后端校验

用于验证数据的合法性、安全性和业务逻辑的正确性；

1. 数据合法性与唯一性校验

需要访问数据库或服务器资源的校验，确保数据的合法性和唯一性：

  • 用户名 / 手机号 / 邮箱唯一性校验：验证用户注册的用户名、手机号、邮箱是否已被注册，需要查询数据库确认。
  • 身份证号合法性校验：验证身份证号的真实性，需要查询公安系统的接口或数据库确认。
  • 邀请码 / 优惠券合法性校验：验证用户输入的邀请码、优惠券是否有效，是否已过期，需要查询服务器数据库确认。

2. 业务逻辑校验

与业务规则相关的校验，确保数据符合业务逻辑：

  • 订单库存校验：提交订单时，验证商品的库存是否充足，需要查询数据库中的库存数据，防止超卖。
  • 支付金额校验：支付时，验证支付金额是否与订单金额一致，防止恶意篡改支付金额。
  • 订单状态校验：验证订单的状态是否允许进行当前操作，比如已完成的订单不允许再次支付。
  • 业务规则校验：比如订单金额必须大于库存成本，组合多个字段的业务规则校验（如活动参与的资格校验，需要验证用户的等级、消费记录等）。

3. 权限与安全校验

验证用户的操作权限和数据安全：

  • 用户权限校验：验证用户是否有执行当前操作的权限，比如只有管理员才能删除用户，普通用户无法访问后台管理页面。
  • 数据归属校验：验证用户是否有权限操作当前数据，比如用户只能修改自己的订单，无法修改他人的订单。
  • 防恶意提交校验：验证请求是否为恶意提交，比如通过 Token 校验、签名校验等方式，防止绕过前端的恶意请求。
  • 数据安全校验：验证提交的数据是否包含恶意内容（如 SQL 注入、XSS 攻击代码），防止恶意攻击。

4. 数据完整性校验

确保提交的数据完整、符合接口要求：

  • 必填字段校验：再次验证表单的必填字段是否存在，防止前端校验被绕过，比如恶意请求中缺失必填字段。
  • 数据格式二次校验：再次验证数据的格式是否正确，比如前端校验被绕过，提交了错误格式的手机号、邮箱。
  • 数据类型校验：验证提交的数据类型是否正确，比如接口要求接收数字类型的金额，防止提交字符串类型的数据。

5. 第三方依赖校验

需要调用第三方接口的校验：

  • 手机号验证码校验：验证用户输入的手机号验证码是否正确，需要调用短信服务商的接口确认。
  • 地址合法性校验：验证用户输入的收货地址是否合法，需要调用地图服务商的接口确认地址的真实性。
  • 支付结果校验：支付完成后，验证支付结果的真实性，需要调用支付服务商的接口确认。

对于实时搜索和按回车（按钮）搜索的区别

实时搜索需要增加一个防抖，也就是当用户未输入时过了300ms后开始执行；

核心结论：多数场景建议加防抖，少数极简场景可省略，移动端输入框的实时校验是否加防抖，取决于校验逻辑的复杂度，具体分析如下：

移动端实时校验的防抖

1. 为什么多数场景需要加防抖？

移动端输入的特性（比如拼音联想输入、手写输入、连续快速打字）会导致 input 事件高频触发（比如输入一个手机号会触发 11 次事件），如果直接在 input 事件里做实时校验：

  • 若校验涉及后端接口请求（比如校验手机号是否已注册）：高频请求会浪费服务器资源，甚至触发接口限流，还可能导致网络请求堆积，影响页面响应；
  • 若校验涉及前端复杂计算（比如密码强度评分、正则多层匹配）：高频计算会增加手机 CPU 消耗，导致页面卡顿、输入不流畅；
  • 无效校验多：比如用户输入手机号时，“138”“1380” 等中间状态的校验无意义，只有用户输入暂停后（比如输完 11 位）的校验才有效。

2. 什么场景可以不加防抖？

仅当校验是纯前端极简逻辑（无接口、无复杂计算）时，可省略防抖，比如：

  • 校验输入是否为空；
  • 校验输入字符长度是否超过限制（如最多 10 位）；
  • 仅校验是否包含数字 / 字母（简单正则）；
  • 仅改变输入框样式（如边框颜色）提示格式问题。

3. 移动端防抖的实用实现示例

防抖的核心是：延迟执行校验逻辑，若在延迟时间内再次触发输入，则重置延迟，直到用户输入暂停后再执行。推荐移动端防抖延迟设为 300~500ms（兼顾体验和性能）。
